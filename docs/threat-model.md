**Методология:** STRIDE  
**Дата:** 2026-06  
**Ссылки:** `TECHNICAL_SPEC.md §3.9`

## 1. Контекст и границы

### 1.1 Что защищаем

- Исходный код и Docker-образы продуктов вендора (IP)
- Лицензионные права: только организации с активным грантом должны использовать продукты
- Данные операторов и телеметрия организаций (персональные данные, бизнес-метрики)
- Доступность: агент должен оставаться функциональным при недоступности Control Plane

### 1.3 Участники и их мотивация

| Актор | Доверие | Мотивация атаки |
|---|---|---|
| Вендор | Полное (владелец CP) | — |
| Оператор (авторизованный) | Высокое (JWT, MFA) | Может ошибиться, insider threat |
| Администратор организации | Среднее (имеет installer) | Может попытаться обойти лицензию |
| Root на хосте организации | Низкое | Может пропатчить бинари |
| Внешний злоумышленник | Нет | MitM, перехват токенов |
| Конкурирующий вендор | Нет | Reverse engineering продуктов |

## 2. Анализ угроз по STRIDE

### 2.1 Spoofing (Подделка идентичности)

#### T-S1: Поддельный агент
- **Угроза:** Злоумышленник регистрирует поддельного агента, получает mTLS-сертификат и Harbor robot credentials, скачивает образы продуктов.
- **Вектор:** Перехват enrollment-токена при передаче оператор → администратор.
- **Контрмеры:**
  - Токен вшит в персонализированный подписанный installer (ADR-0004), не передаётся отдельно.
  - Enrollment-токен одноразовый (переходит в статус `Used` после использования).
  - Harbor robot account scoped: только к продуктам с активным грантом.
- **Остаточный риск:** Низкий. Атака требует физического доступа к installer-архиву и его декомпиляции.

#### T-S2: Поддельный Control Plane
- **Угроза:** Злоумышленник поднимает fake agent-gateway, перенаправляет агент через DNS poisoning.
- **Вектор:** DNS MitM в сети организации.
- **Контрмеры:**
  - Агент пиннет CA-сертификат вендора при enrollment.
  - TLS с проверкой cname + CA chain.
- **Остаточный риск:** Низкий при корректной реализации pinning.

#### T-S3: Подделка оператора (stolen JWT)
- **Угроза:** Злоумышленник получает JWT оператора, управляет грантами и агентами.
- **Контрмеры:**
  - JWT короткоживущий (1-4 ч) с refresh.
  - MFA обязательна для операторов (§3.1).
  - Audit log всех действий (топик `audit.operator.action`).
- **Остаточный риск:** Средний. При компрометации refresh-токена — ручной отзыв через auth-service.

---

### 2.2 Tampering (Нарушение целостности)

#### T-T1: Подмена Docker-образа продукта
- **Угроза:** Администратор организации или MitM подменяет образ на вредоносный.
- **Вектор:** Man-in-the-middle между Harbor и агентом; патч образа на диске.
- **Контрмеры:**
  - Все образы имеют cosign-подпись (§3.6).
  - Агент верифицирует подпись перед `docker compose up`.
  - Digest (`sha256:...`) фиксируется в compose-шаблоне, Docker проверяет при pull.
- **Остаточный риск:** Низкий при enforcement верификации cosign.

#### T-T2: Патч License SDK в бинаре сервиса
- **Угроза:** Root на хосте патчит бинарь сервиса, отключая проверку лицензии.
- **Контрмеры:**
  - License SDK — двойной слой: `license-sidecar` + SDK в каждом сервисе. Удаление sidecar не помогает.
  - Self-integrity check в SDK (hash own binary at startup).
  - Cosign-проверка образа при старте контейнера (возможность).
- **Остаточный риск:** Высокий. 100% защита кода на чужом железе недостижима. Признан в §2.15 и ADR-0003.

#### T-T3: Изменение compose-шаблона
- **Угроза:** Организация изменяет docker-compose.yml, убирает license-sidecar или меняет конфиг.
- **Контрмеры:**
  - Агент применяет только compose-шаблоны из каталога (подписанные).
  - Шаблон доставляется через защищённый канал (mTLS) и кэшируется с верификацией.
  - Запуск compose вручную (в обход агента) — остаточный риск; закрывается двойным лицензированием (T-T2).
- **Остаточный риск:** Средний (запуск вручную возможен).

### 2.3 Repudiation (Отрицание действий)

#### T-R1: Оператор отрицает действие с грантом
- **Угроза:** Оператор отзывает грант, затем утверждает что не делал этого.
- **Контрмеры:**
  - Все действия операторов публикуются в топик `audit.operator.action` с immutable retention (365 дней).
  - Аудит-записи содержат JWT-клеймы (`actorId`, `ipAddress`, `timestamp`).
  - Эндпоинт `GET /v1/audit` с фильтрацией и экспортом.
- **Остаточный риск:** Низкий.

---

### 2.4 Information Disclosure (Утечка информации)

#### T-I1: Утечка Harbor robot password
- **Угроза:** Robot-пароль Harbor, выданный при enrollment, утекает с хоста агента.
- **Вектор:** Компрометация хоста, чтение конфигов агента.
- **Контрмеры:**
  - Пароль хранится в keychain или зашифрованном файле (не в plaintext env).
  - Robot account scoped только к продуктам с активным грантом.
  - При отзыве гранта fleet-service убирает scope → robot не имеет доступа к pull.
- **Остаточный риск:** Средний. Ротация robot credentials по расписанию — рекомендация (§2.7 Q8).

#### T-I2: Утечка метрик и телеметрии организации
- **Угроза:** Данные хоста (CPU, RAM) и метрики продуктов видны несанкционированным операторам.
- **Контрмеры:**
  - Авторизация через JWT с `tenantId` клеймом.
  - RBAC: оператор видит только свои организации.
  - Данные изолированы по tenantId в БД (§3.3).
- **Остаточный риск:** Низкий.

#### T-I3: Реверс-инжиниринг Docker-образов
- **Угроза:** Организация экспортирует образ и изучает исходный код продукта.
- **Контрмеры:**
  - Обфускация бинарей (не описана в спеке — рекомендация).
  - Образы хранятся только в Harbor с RBAC; pull закрывается при истечении гранта.
- **Остаточный риск:** Высокий. Принципиально: code on untrusted hardware = full access.

---

### 2.5 Denial of Service (Отказ в обслуживании)

#### T-D1: DDoS на agent-gateway
- **Угроза:** Злоумышленник или скомпрометированный агент флудит агент-gateway.
- **Контрмеры:**
  - Rate limiting на agent-gateway (§2.14).
  - mTLS обязателен — нельзя атаковать без сертификата.
  - Circuit breaker перед внутренними сервисами.
- **Остаточный риск:** Средний (легитимные агенты с сертификатом могут флудить).

#### T-D2: Недоступность Control Plane → HardStop продукта
- **Угроза:** Агент не может обновить лиз (CP недоступен) → продукт переходит в HardStop.
- **Контрмеры:**
  - Grace period 7 дней: продукт работает без лиза `TTL + 7 дней`.
  - Лиз кэшируется локально на диске агента.
  - Circuit breaker + stale cache на agent-gateway.
- **Остаточный риск:** Низкий для кратких недоступностей (< 7 дней + TTL).

#### T-D3: Kafka partition lag — задержка обработки grant status changed
- **Угроза:** fleet-service не успевает обновить Harbor scope при пиковой нагрузке.
- **Контрмеры:**
  - Kafka retention 30 дней — события не теряются.
  - Monitoring: consumer lag алерт при > 1000 сообщений.
  - Agent дополнительно обнаруживает исчезнувший грант через polling (pull-based fallback).
- **Остаточный риск:** Низкий.

---

### 2.6 Elevation of Privilege (Повышение привилегий)

#### T-E1: Агент с Docker socket = root на хосте
- **Угроза:** Компрометация агента-процесса даёт root на хосте через Docker socket.
- **Контрмеры:**
  - Агент запускается от минимальных привилегий (non-root), только socket mount.
  - Rootless Docker рекомендован для продакшен-инсталляций.
  - AppArmor/seccomp профили для Docker daemon.
- **Остаточный риск:** Высокий. Фундаментальное ограничение архитектуры (признано в §1.2).

#### T-E2: Оператор создаёт себе суперправа через API
- **Угроза:** Оператор с правами на создание продуктов создаёт себе неограниченный грант.
- **Контрмеры:**
  - RBAC разграничивает: `catalog-manager`, `fleet-manager`, `billing-admin` (§3.1).
  - Принцип наименьших привилегий — ни одна роль не имеет полного доступа.
  - Audit log каждого действия.
- **Остаточный риск:** Низкий при корректной RBAC-модели.

## 4. Принятые риски и обоснование

| Риск | Обоснование принятия |
|---|---|
| T-T2 (патч бинаря) | 100% защита кода на чужом железе недостижима. Двойной слой (sidecar + SDK) максимизирует стоимость атаки. |
| T-I3 (реверс образов) | Неизбежно при on-prem модели. Обфускация повышает планку, не устраняет риск. |
| T-E1 (Docker socket = root) | Фундаментальное требование архитектуры. Rootless Docker и AppArmor снижают риск. |

## 5. Рекомендации для будущих итераций

1. **Rootless Docker** как требование для продакшен-инсталляций (сейчас рекомендация).
2. **Обфускация бинарей** продуктовых образов (GoGaObfuscate для Go, ProGuard для Java).
3. **Robot credentials rotation** по расписанию (Q8 в §2.16).
4. **Signed URL для installer** (Q9 в §2.16) — 4-часовая ссылка вместо прямой передачи файла.
5. **Sidecar as reverse proxy** (ADR-0003, эволюция) — весь трафик через sidecar с проверкой лиза.
