# ADR-0001: Агент-апплайанс написан на Go, а не на Java/Spring

**Статус:** Принято  
**Дата:** 2026-06  
**Ссылки:** `TECHNICAL_SPEC.md §2.1`, `§2.8`

---

## Контекст

Агент-апплайанс — демон на сервере клиента, который:
- запускается как systemd-юнит при старте ОС,
- управляет Docker Engine через `/var/run/docker.sock`,
- работает даже при недоступности Control Plane,
- поставляется на Linux x86_64/arm64 без предустановленного JVM.

Весь остальной Control Plane написан на Java/Spring Boot. Рассматривался вопрос: не использовать ли Spring Boot и для агента (в виде Docker-контейнера)?

---

## Рассмотренные варианты

### A. Go — статический бинарь + systemd (выбрано)

- Компилируется в один статический бинарь: `GOOS=linux GOARCH=arm64 go build`.
- Стартует за < 1 с; не требует JVM.
- Запускается как systemd-юнит напрямую — не зависит от Docker daemon.
- Потребление памяти в покое: ~30–50 MB.

### B. Java/Spring Boot — в Docker-контейнере

- Требует Docker для запуска, но агент сам управляет Docker. Возникает **bootstrap-проблема**: кто запустит контейнер агента, если Docker упал?
- Для управления Docker-сокетом из контейнера нужно монтировать `/var/run/docker.sock` — это даёт контейнеру **полный root на хосте** (стандартный вектор эскалации привилегий).
- JVM + Spring Boot: ~300–500 MB RAM, старт 5–15 с.
- Привычный стек для команды, разрабатывающей Control Plane.

---

## Решение

**Вариант A — Go.**

---

## Последствия

**Позитивные:**
- Агент независим от Docker daemon; может перезапустить его сам через systemd.
- Нет монтирования docker.sock в контейнер — сниженная поверхность атаки.
- Один бинарь для x86_64 и arm64 без отдельных JVM-образов.
- Меньший ресурсный след на хосте клиента.

**Негативные:**
- Отдельная кодовая база на Go: другие инструменты, CI-пайплайн, тесты.
- Разработчики Control Plane (Java) не пишут агента без дополнительного onboarding.

**Примечание для учебной версии:**  
В `EDU_PROJECT_PLAN.md` агент намеренно заменён на Spring Boot-сервис для педагогических целей — чтобы не вводить Go в учебный проект. Это учебное упрощение, не изменение архитектурного решения.
