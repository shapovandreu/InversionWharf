# ADR-0003: Двухслойное рантайм-лицензирование — sidecar + SDK

**Статус:** Принято с оговорками  
**Дата:** 2026-06  
**Ссылки:** `TECHNICAL_SPEC.md §2.15`, `§3.7`, `§3.9`

---

## Контекст

Бизнес-требование `B9`: продукт должен прекращать обслуживание после истечения льготного периода (7 дней) без действующего лицензионного лиза. Это должно работать даже если организация запускает образы напрямую, минуя агента.

Нужно выбрать: где и как физически реализовать принуждение.

---

## Рассмотренные варианты

### A. Только sidecar-контейнер (без SDK в сервисах)

- `license-sidecar` получает лиз и кладёт его в shared-volume.
- Сервисы читают лиз из volume и... проверяют его самостоятельно? Без SDK — нет.
- Вариант «sidecar как reverse proxy»: весь трафик к сервисам идёт через sidecar, который проверяет лиз и блокирует при HardStop.

**Плюсы:** один компонент для обновления; языконезависимо.  
**Минусы:** убрать sidecar из `docker compose up` → продукт работает без ограничений. Защита только через cosign-подпись compose-шаблона (агент отвергает изменённый шаблон), но организация может запустить compose вручную.

### B. Sidecar + SDK в каждом сервисе (выбрано)

- `license-sidecar`: продление лиза через агента, кэш в shared-volume (read-only).
- `License SDK` (Go/Java/Node): встроен в каждый сервис, офлайн-проверка подписи лиза пиннутым публичным ключом, применение состояний `Valid/Grace/HardStop`.

**Плюсы:** удаление sidecar без SDK не помогает — сервисы сами уходят в HardStop после `TTL+grace`. Принуждение двухслойное.  
**Минусы — операционные:** поддержка SDK на каждом языке; пересборка всех образов при обновлении SDK; ротация пиннутого публичного ключа = каскадная пересборка.

---

## Решение

**Вариант B** — двухслойный подход, как более стойкий к обходу.

---

## Оговорки и операционные риски

**Ротация ключа подписи (критический риск):**  
Публичный ключ вендора пиннут в SDK. При ротации (компрометация или плановая):
1. Обновить SDK во всех языках.
2. Пересобрать все продукты с новым SDK.
3. Переподписать все образы.
4. Клиенты должны обновиться до ротации ключа на `license-service`.

Процедура описана в [`docs/key-rotation.md`](../key-rotation.md).

**Мультиязычный SDK:**  
Требует поддержки Go, Java, Node (минимум). Каждое изменение поведения — тройное изменение. Рекомендация: выделить SDK в отдельный репозиторий с матрицей тестирования.

**Остаточный риск (признан в `§2.15`):**  
Root на хосте может пропатчить бинарь сервиса. Контрмеры повышают планку (обфускация, self-integrity-check, cosign-проверка образа при старте), но 100% защита кода на чужом железе недостижима.

---

## Возможная эволюция

При накоплении опыта поддержки SDK рассмотреть гибрид: sidecar как reverse proxy + минимальный SDK только для startup-check. Это снижает нагрузку на мультиязычную поддержку при сохранении базового принуждения.
