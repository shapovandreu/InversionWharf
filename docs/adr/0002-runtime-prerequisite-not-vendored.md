# ADR-0002: Docker Engine как предусловие, а не вендоринг внутри installer

**Статус:** Принято  
**Дата:** 2026-06  
**Ссылки:** `TECHNICAL_SPEC.md §1.1`, `§1.2`, `§2.8`, `§2.14`

---

## Контекст

Агент-апплайанс управляет Docker-контейнерами продуктов через Docker API. Изначально спецификация (до v4.1) описывала агент как «самодостаточный пакет со встроенной контейнерной средой» — installer вендорил `dockerd` + `containerd` + Compose v2 пиннутых версий. Организации не требовался предустановленный Docker.

Рассматривался вопрос: стоит ли вендорить runtime или требовать его как предусловие?

---

## Рассмотренные варианты

### A. Вендоренный Docker Engine (исходный подход)

- Installer распаковывает и запускает собственный `dockerd` + `containerd`.
- Нет зависимости от хостового Docker.
- **Проблемы:**
  - Конфликты с уже установленным Docker на хосте (cgroups, namespaces, socket-пути).
  - Необходимость поддерживать совместимость с разными ядрами Linux и дистрибутивами.
  - Самостоятельный security-патчинг вендоренного runtime.
  - Installer ~500 MB против ~5 MB для Go-демона.
  - Фактически — конкурент Docker Desktop, что не является целью.

### B. Docker Engine ≥ 24 как предусловие хоста (выбрано)

- Installer проверяет `docker version` и `docker compose version`; завершается с диагностикой если не удовлетворяет.
- Агент работает через `/var/run/docker.sock` хостового Docker.
- **Обоснование:** целевая аудитория — on-prem серверы организаций, поставляющих Docker-продукты. Docker Engine на таких серверах уже есть практически всегда.

### C. containerd напрямую (без Docker Engine)

- Меньший размер, Apache 2.0.
- Несовместимо с compose-шаблонами из каталога без `nerdctl compose`.
- Потребовало бы замены всей compose-экосистемы.

---

## Решение

**Вариант B — Docker Engine ≥ 24 (или containerd ≥ 1.7 + Compose v2) как предусловие хоста.**

---

## Последствия

**Позитивные:**
- Installer уменьшается с ~500 MB до ~5 MB.
- Нет конфликтов с хостовым Docker; не нужно поддерживать вендоренный runtime.
- Security-патчи Docker применяет сама организация через штатные механизмы (apt/yum).
- `RuntimeManager` в агенте упрощается до проверки socket и управления compose-проектами.

**Негативные:**
- Организация обязана установить Docker до установки апплайанса — дополнительный шаг onboarding.
- Нельзя гарантировать точную версию Docker на хосте (только минимальную).
- При поломке хостового Docker агент тоже теряет функциональность (но не падает как процесс).

**Документирование в спеке:**  
`§2.14` NFR: «Совместимость хоста: Linux x86_64/arm64, systemd, Docker Engine ≥ 24 или containerd ≥ 1.7».
