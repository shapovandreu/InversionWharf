# ADR-0004: Enrollment-токен вшит в персонализированный installer

**Статус:** Принято  
**Дата:** 2026-06  
**Ссылки:** `TECHNICAL_SPEC.md §2.7 fleet-service`, `§2.8`, `§2.9 S1`, `§3.9`

---

## Контекст

При регистрации агента оператор создаёт агента в системе и получает одноразовый `enrollmentToken`. Этот токен нужно передать администратору организации, который устанавливает апплайанс на сервере клиента.

Вопрос: **как безопасно передать токен** от оператора к администратору организации?

---

## Рассмотренные варианты

### A. Отдельный токен (исходный подход)

Оператор получает `{agentId, enrollmentToken, ссылка на installer}` и передаёт токен + ссылку администратору любым каналом (email, Jira, мессенджер).

**Проблема:** токен — секрет. Передача по email/чату:
- Может быть перехвачена (MitM, утечка из почтового ящика).
- Остаётся в истории переписки после использования.
- Кто перехватил токен — может зарегистрировать поддельного агента и получить mTLS-сертификат + Harbor robot-credentials.

### B. Однократный показ токена в Dashboard (одноразово как SSH-ключ)

Токен показывается в UI один раз сразу после генерации. Оператор копирует вручную.

**Плюс:** токен не ходит по email.  
**Минус:** всё равно нужно передать его администратору каким-то каналом. Проблема сдвигается, а не решается.

### C. Токен вшит в персонализированный installer (выбрано)

Dashboard при создании агента генерирует **персонализированный подписанный архив**: стандартный installer + токен, упакованный внутри (не в открытом виде в отдельном файле). Оператор передаёт архив администратору любым каналом.

- Токен защищён подписью архива: его нельзя извлечь без распаковки и патча бинаря.
- Канал передачи архива может быть любым (email, SFTP) — даже если архив перехвачен, токен не виден без декомпиляции.
- Enrollment запускается автоматически при первом старте — администратору не нужно ничего вводить.

### D. Короткоживущая HTTPS-ссылка для скачивания

Dashboard генерирует signed URL (например, 4-часовой) для скачивания персонализированного installer. Администратор открывает ссылку в браузере.

**Плюс:** ещё безопаснее — ссылка сгорает.  
**Минус:** требует доступа организации к HTTPS-эндпоинту CP в момент установки; усложняет flow при air-gapped инсталляциях.

---

## Решение

**Вариант C** — токен вшит в персонализированный подписанный архив.

Вариант D можно реализовать поверх C как улучшение: ссылка ведёт к скачиванию персонализированного архива. Фиксируется как `Q9` в `§2.16` для будущего рассмотрения.

---

## Последствия

**Позитивные:**
- Токен не фигурирует отдельно ни в каком канале связи.
- Enrollment полностью автоматизирован: запустил installer → агент сам зарегистрировался.
- Упрощённый UX для администратора организации.

**Негативные:**
- Каждый агент требует генерации отдельного installer (не один универсальный).
- fleet-service должен уметь генерировать подписанные архивы — дополнительная сложность.
- Размер передаваемого файла чуть больше (installer + вшитый токен).

**Влияние на бэклог:**  
`JIRA_BACKLOG.md S-4.2` (Enrollment по одноразовому токену) — нужно уточнить `T-4.2.1`: вместо «хранение `хеш_токена`» описать генерацию персонализированного архива.
