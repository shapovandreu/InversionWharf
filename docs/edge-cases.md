# Edge Cases и Failure Scenarios

**Ссылки:** `TECHNICAL_SPEC.md §2.9`, `§2.15`, ADR-0003

---

## 1. Offline Agent (CP недоступен)

### Сценарий
Агент не может подключиться к agent-gateway (сетевой сбой, плановое обслуживание CP, DNS-проблема).

### Поведение

| Период | Состояние агента | Состояние продукта |
|---|---|---|
| 0 – TTL лиза (до 72 ч) | offline, лиз валиден | `licenseState: Valid` — работает нормально |
| TTL < t < TTL + 7 дней | offline, лиз истёк | `licenseState: Grace` — работает с предупреждением |
| t > TTL + 7 дней | offline | `licenseState: HardStop` — продукт прекращает обслуживание |

### Что делает агент
1. При каждом heartbeat retry — exponential backoff с jitter (1s, 2s, 4s, 8s, max 5 мин).
2. После 3 неудачных попыток — логирует ошибку, но не падает.
3. Lease cache (локальный файл) читается License SDK при каждой startup-проверке.
4. При восстановлении связи — немедленный heartbeat + продление лиза.

### Граничные случаи
- **Лиз истёк во время offline + восстановление < 7 дней:** продукт в Grace, лиз обновляется сразу при reconnect.
- **Восстановление после 7+ дней:** продукт в HardStop, лиз обновляется, но продукт не возобновляется автоматически — нужен рестарт контейнеров после получения нового лиза.
- **Лиз истёк строго в момент деградации CP (race condition):** grace period даёт буфер 7 дней, исключая ложный HardStop при кратких сбоях CP.

## 2. Enrollment в Air-Gapped среде

### Сценарий
Организация устанавливает апплайанс на сервер без доступа к интернету.

### Проблема
Enrollment требует HTTP-соединения с agent-gateway для `/v1/enroll`.

### Текущее поведение
Enrollment невозможен без доступа к agent-gateway — это известное ограничение (§2.16 Q9).

### Обходные пути (не реализованы)
1. **Signed URL с прокси:** организация скачивает installer через интернет-шлюз, затем переносит на air-gapped сервер.
2. **Offline enrollment package:** vault PKI генерирует сертификат заранее, вкладывается в installer вместе с токеном. Зафиксировано как `Q9` для будущего рассмотрения.

## 3. Истечение mTLS-сертификата агента

### Сценарий
Vault PKI выдаёт mTLS-сертификат с TTL (например, 90 дней). Агент не обновил сертификат до истечения.

### Поведение
- Агент пытается подключиться к agent-gateway → TLS handshake fail (expired cert).
- agent-gateway возвращает 401 (отозванный/истёкший сертификат).
- Агент не может обновить лиз → grace period начинается.

### Процедура восстановления
1. Проверить дату истечения: `openssl x509 -in /etc/inversionwharf/agent.crt -noout -dates`
2. Если истёк — выполнить re-enrollment (см. [runbooks/re-enroll.md](runbooks/re-enroll.md)).

### Превентивные меры
- Агент проверяет TTL сертификата при старте; если < 14 дней до истечения — начинает auto-renew через `/v1/enroll` (только если текущий сертификат ещё действует).
- Алерт: `cert_expiry_days < 30` → PagerDuty / Slack.

## 4. Grant Revoked пока агент offline

### Сценарий
Оператор отзывает грант продукта. В это время агент offline (например, плановое обслуживание сервера).

### Последовательность событий
1. Оператор отзывает грант в Dashboard → entitlement-service публикует `entitlement.grant.status.changed`.
2. fleet-service убирает scope Harbor robot (pull закрыт мгновенно для новых pull).
3. Агент offline — не получает сигнал.
4. Kafka-событие хранится (retention 30 дней).
5. Агент восстанавливается, делает `GET /v1/products` → продукт отсутствует в ответе (грант отозван).
6. `CatalogPoller` обнаруживает исчезнувший продукт → сигнал `CleanupManager`.
7. `CleanupManager` вызывает `docker image rm` для образов продукта.
8. При попытке продления лиза → 403 LICENSE_DENIED.

### Граничные случаи
- **Агент offline > 7 дней:** продукт сам уходит в HardStop по истечении grace. При reconnect cleanup также выполняется.
- **Агент offline и образы уже удалены (docker system prune):** `docker image rm` вернёт not found — агент логирует и продолжает.

## 5. Duplicate Enrollment (токен использован дважды)

### Сценарий
Персонализированный installer скопирован и запущен на двух серверах.

### Поведение
- Первый агент выполняет `/v1/enroll` → токен переходит в `Used`, агент получает сертификат.
- Второй агент пытается `/v1/enroll` с тем же токеном → `409 Conflict` (enrollment-token-already-used).
- Второй агент не регистрируется, не получает сертификат.

### Как обнаружить
- Оператор видит в Dashboard: агент зарегистрирован с неожиданного IP.
- Audit log: `ENROLLMENT_ATTEMPTED` с `status: failed`, IP второго сервера.

### Процедура
1. Оператор деактивирует зарегистрированного агента (если IP неизвестен).
2. Создаёт нового агента → новый персонализированный installer.
3. Устанавливает на корректный сервер.

## 6. License Sidecar Crash

### Сценарий
`license-sidecar` контейнер крашится (OOM, runtime error). Агент не перезапускает его (не управляет lifecycle контейнеров напрямую — только через compose).

### Поведение
- `license-sidecar` падает → compose restart policy (`restart: always`) перезапускает его.
- Между падением и рестартом: lease cache на shared-volume ещё актуален.
- License SDK в сервисах читает кэш из volume напрямую — работает независимо от sidecar.
- Если restart loop (crashloop): лиз не обновляется → через TTL начинается grace period.

### Граничные случаи
- **Shared volume недоступен (corrupted):** License SDK не может прочитать лиз → немедленный переход в Grace (не HardStop, так как ошибка чтения ≠ истёкший лиз).
- **Sidecar крашится при enrollment (нет кэша):** продукт стартует в Grace с нулевым кэшем; если лиз не получен до истечения 7 дней — HardStop.

## 7. Rollout Failure (Partial Upgrade)

### Сценарий
Агент запускает обновление продукта (`docker compose pull && docker compose up -d`). Один из образов не скачивается (Harbor недоступен, сеть рвётся).

### Поведение
- `docker compose pull` завершается с ошибкой для одного образа.
- Агент НЕ запускает `docker compose up -d` (atomicity: pull-or-nothing).
- Старая версия продолжает работать.
- Агент логирует ошибку и повторяет попытку через 5 мин (exponential backoff).
- Installation report: `health: Failed`, `version: <old>` (обновление не применено).

### Граничные случаи
- **Partial pull (один образ скачан, второй нет):** Docker не заменяет рабочие контейнеры — старые образы используются. При следующем pull Docker перекачивает только недостающее (layer cache).
- **Cosign verify fail для нового образа:** агент логирует security error, не применяет обновление. Требует проверки оператором.

## 8. Clock Skew (Рассинхронизация времени)

### Сценарий
Хост агента имеет значительный clock skew (NTP не настроен) — например, +30 мин.

### Влияние на лицензирование
- JWT-лиз содержит `exp` (absolute timestamp). License SDK проверяет `exp` по локальному времени.
- **Если хост опережает:** лиз кажется истёкшим раньше → преждевременный Grace.
- **Если хост отстаёт:** лиз кажется действительным дольше → можно exploit.

### Контрмеры
- License SDK допускает clock skew tolerance ± 5 минут при проверке `exp`.
- Installer проверяет синхронизацию времени (NTP check) перед enrollment.
- NFR: NTP обязателен на хосте (§2.14).

## 9. Harbor Rate Limiting

### Сценарий
Все агенты одновременно пытаются pull новые образы после публикации релиза.

### Поведение
- Harbor может применять rate limiting → `429 Too Many Requests`.
- Агент использует exponential backoff + jitter при `429`.
- `docker pull` повторяет попытку автоматически.

### Контрмеры
- `catalog.release.published` → агенты получают событие с jitter (agent-gateway добавляет random delay 0-30 мин).
- Harbor bandwidth throttling per robot account.
