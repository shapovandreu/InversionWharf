# InversionWharf — Backlog для Jira

> Иерархия для импорта в Jira: **Эпик → История (Story) → Задача (Task) → Подзадача (Sub-task)**.
> Каждый элемент ссылается на возможности (`B1…B9`), сценарии (`S1…S7`) и параграфы (`§…`) из `TECHNICAL_SPEC.md`.
> Ключи условные (`EPIC-n` / `S-n.m` / `T-n.m.k` / `ST-…`) — при импорте заменятся на ключи проекта (например `IW-…`).

## Условные обозначения
- **Story** оформляется как «Как `<актор>`, я хочу `<…>`, чтобы `<…>`».
- Каждая Story несёт критерии приёмки (**AC**).
- Акторы — из §1.6: Оператор релизов, Админ организации, Агент, Сервис.
- Оценка (SP, Story Points) — указана в заголовке каждой истории по ряду Фибоначчи (1/2/3/5/8/13); ориентир для планирования, не обязательство.

## EPIC-1 — Платформенная инфраструктура и окружения

- **T-1.1.2** — Helm-чарты-шаблоны для Spring-сервисов
  - ST-1.1.2.1 — Базовый chart (values, probes, ресурсы, HPA)

## Рекомендованная последовательность реализации
Порядок выстроен по зависимостям: сначала фундамент (инфраструктура, периметр, сквозные механизмы), затем базовые домены, затем сквозной поток доставки, после — лицензирование, и в конце — операционная зрелость. **UI и наблюдаемость распределены по всем фазам**: как только готов backend-домена, к нему добавляется UI-срез — так в конце каждой фазы есть что показать заказчику. Внутри фазы истории можно вести параллельно.

| Фаза | Цель | Истории (в порядке старта) | UI-срез фазы (демо) | Ключевые зависимости |
| --- | --- | --- | --- | --- |
| **0. Фундамент** | Воспроизводимое окружение и пайплайн | S-1.1 → S-1.2 → S-1.3 → S-13.1 (минимально) | — (инфраструктура) | — |
| **1. Периметр и сквозное** | Идентичность, gateway, общие контракты, каркас UI | S-2.1 → S-2.4 → S-12.2 → S-12.1 → S-2.2 → S-2.3 → S-12.3 → S-11.1 → **S-10.1** | Вход в Dashboard (OIDC), навигация, заглушки доменов | Фаза 0; S-10.1 требует S-2.2 (BFF) |
| **2. Базовые домены** | Каталог и флот + их UI | S-3.1 → S-9.1 → S-3.2 → S-3.3 → S-3.4 → S-4.1 → S-4.2 → S-4.3 → S-9.2 → **S-10.2** → **S-10.3** | Публикация релиза, промоушн по каналам, регистрация агента — из UI | Фаза 1; S-3.2 требует S-9.1; S-4.2 требует Vault PKI (S-1.2); S-10.2/10.3 — поверх catalog/fleet API |
| **3. Сквозная доставка (MVP)** | Гранты + агент + отчётность + их UI | S-5.1 → S-5.2 → S-8.1 → S-8.2 → S-8.3 → S-8.4 → S-7.1 → S-7.2 → S-7.3 → S-8.6 → **S-10.4** → **S-10.5** | Выдача гранта → установка на агенте → отчёт и дашборды здоровья/аудита в UI | S-5.2 требует S-3.4; S-8.2 требует S-4.2; S-8.4 требует S-9.1; S-10.4/10.5 — поверх grants/telemetry API |

| **5. Операционная зрелость** | Финализация наблюдаемости и дистрибуции | S-11.2 → S-13.2 | Дашборды Grafana, алертинг; подписанные пакеты агента | Поверх фаз 1–4 |

**Контрольная точка MVP — после фазы 3:** работает онлайновая pull-доставка (`B1`–`B7`) — публикация и продвижение релизов, регистрация агентов, гранты, ручная установка/обновление с проверкой целостности и телеметрией, **с операторским UI на каждый из этих шагов**. Рантайм-лицензирование (`B9`) добавляется в фазе 4 как усиление, не блокируя базовый сценарий доставки.

**Демонстрируемый результат по фазам:**
- **Фаза 1** — рабочий вход и каркас Dashboard (S-10.1).
- **Фаза 2** — управление каталогом и флотом из UI (S-10.2, S-10.3).
- **Фаза 3** — полный сценарий доставки с грантами, телеметрией и аудитом в UI (S-10.4, S-10.5).
- **Фаза 4** — лицензирование и отчёт «Льготный период» в UI (S-10.6).
